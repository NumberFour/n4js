Release Notes

The current release of N4JS, version 0.3, is a "public alpha". "alpha" means four things:

  1. There exist known issues
  2. There exist probably even more unknown issues.
  3. There will be more features in future releases.
  4. There may be incompatible changes in future releases.

In case of N4JS "alpha" does not mean unstable. N4JS has been developer for three years now. There exists an extensive test suite with more that 70.000 tests, including the ECMAScript test suites. It is used in production mode internally for more than a year, handling a code base with more than 3.000 modules and thousands of tests. That is N4JS has successfully passed its internal beta phase: It is working for us. So we are confident that it will also work for you. But since every programmer has her/his own style, you may find issues we didn't find. In order to solve these issues, we may add new features or have to change things. This is why we call it "public alpha".

Known Issues

Since we are using N4JS and the N4JS IDE internally for quite a while we have found a lot of problems of course. There are still a lot of bugs open, and we have lots of ideas for new features. We used an non-public bug tracking system to manage bugs internally. We will review these issues and migrate them to GitHub issues as soon as possible.

Unkown Issues

We embrace bug reports and feature requests!

For the moment we use GitHub issues to manage issues found by public users. When we will move to a different bug-tracking system, we will try to migrate these issues.

Writing bug reports is not easy. In order to help you describing bugs and new features, the N4JS IDE comes with an extra language especially designed to enable users of a N4JS to write good bug reports.

New Features

In the future, the following topics are expected to be addressed:

  1. Improved UI experience, in particular customized content assist as well as more quick fixes.
  2. Improved ECMAScript 2015 support.
  3. Improved type inference and add type guards.
  4. Improved refactoring capabilities.
  5. Improved node.js developer experience.
  6. Support for browser-based projects.

There may be other topics raised by users. Breaking down these topics into single features and prioritizing them highly depends on your feedback!

Incompatible Changes in Future Releases

Yes, that may happen. Maybe the language or APIs have to be changed in future releases, breaking existing code. But don't worry! As mentioned afore, we have a large internal code base. So if incompatible changes are introduced, we will provide tools to migrate your code automatically (unless the changes are easier to solve manually).

Just to given you an idea: In earlier versions we used a Java-like syntax for type annotations. Since this did not work well together with other features such as async/await, we changed the syntax to ECMAScript 4 colon style. This required our whole code base to be changed. Fortunately this was easy since we provided a quick fix, enabling almost all problems stemming from that change to be fixed with a single key stroke!
